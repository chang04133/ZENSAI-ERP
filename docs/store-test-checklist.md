# 매장 테스트 체크리스트

> **목적**: 시스템 업데이트 후 매장에서 실제 운영 기능이 정상 동작하는지 확인
> **대상**: 매장관리자(STORE_MANAGER), 매장직원(STORE_STAFF)
> **테스트 환경**: `npm run dev:store` (포트 5174, gangnam 계정) / `npm run dev:staff` (포트 5175, daegu 계정)
> **방법**: 아래 항목을 순서대로 실행하고, 문제 발견 시 항목 번호와 증상을 공유
> **참고**: ✅ 예상 결과를 반드시 확인. 하나라도 다르면 버그 보고

---

## A. 로그인 / 기본 접속

### A-1. 로그인 → 대시보드 표시
- **조작**: 아이디/비밀번호 입력 → 로그인 버튼
- **예상 결과**:
  - 대시보드 페이지(`/`)로 이동
  - 상단에 보라색 또는 빨간색 히어로 배너 표시
  - "오늘 매출" 카드(보라색 그라데이션)에 금액 표시
  - 하단에 "오늘 판매 내역" 테이블 표시
- ☐ 확인

### A-2. 좌측 메뉴 — 역할에 맞는 메뉴만 표시
- **조작**: 좌측 사이드바 메뉴 전체 확인
- **매장관리자 예상 메뉴**:
  - 대시보드, 공지사항, 바코드관리
  - 상품관리 > 상품목록
  - 재고관리 > 재고현황, 매장별 재고, 매장 재입고 추천
  - 입고관리 > 종합입고관리, 입고조회
  - 출고관리 > 종합출고관리, 반품관리, 수평이동, 출고조회
  - 판매관리 > 종합매출현황, 판매분석, 매출관리, 고객반품관리
  - 고객관리
  - 직원관리
  - 활동 로그
- **보이면 안 되는 메뉴**: 재고조정, 재고처리, 재고변동내역, 행사관리, 악성재고, 판매율분석, 예약판매(별도메뉴), 거래처관리, 코드관리, 시스템관리, 자금관리, 생산기획
- ☐ 확인

### A-3. 우측 상단 사용자 정보
- **조작**: 우측 상단 헤더 영역 확인
- **예상 결과**: 로그인한 이름 + 소속 매장명(예: "강남점") 표시
- ☐ 확인

### A-4. 내 정보 수정 + 비밀번호 변경
- **조작**:
  1. 우측 상단 이름 클릭 → "내 정보" 페이지(`/my-profile`)
  2. 이름 확인 (읽기 전용: 아이디, 직급, 소속)
  3. "새 비밀번호" 입력 (4자 이상) + "비밀번호 확인" 입력
  4. 저장 → 로그아웃
  5. 새 비밀번호로 재로그인
- **예상 결과**:
  - 저장 시 "수정되었습니다" 성공 메시지
  - 새 비밀번호로 정상 로그인
  - 비밀번호 불일치 시 "비밀번호가 일치하지 않습니다" 에러
- ☐ 확인

## B. 대시보드

### B-1. 대시보드 카드 — 매출/재고/할일 표시
- **조작**: 대시보드 페이지 접속 (`/`)
- **예상 결과**:
  - **히어로 배너**: 미처리 건수 있으면 빨간 배경 + "N건" 표시, 없으면 보라색 배경
  - **할일 카드들** (해당 건수 > 0일 때만 표시):
    - 수령 대기 (초록) — 본사에서 보낸 출고 중 SHIPPED 상태
    - 출고 처리 (주황) — APPROVED 상태 출고 중 우리 매장이 출발지인 것
    - 수량불일치 (빨강) — DISCREPANCY 상태
    - 예약판매 대기 (노란) — 미해소 예약판매 건수
    - 수평이동 출고/수령대기 (청록)
  - **오늘 매출 카드**: 오늘 매출 금액, 판매 수량, 판매 건수
  - **매출 등록 버튼**: 흰색 큰 버튼
  - **오늘 판매 내역 테이블**: 최근 20건 (시간, 상품명, 유형 태그, 수량, 금액)
  - **재고 부족 알림**: 부족 상품 있으면 테이블 표시 (상품명, SKU, 재고, 다른 매장 재고)
- ☐ 확인

### B-2. 미처리 알림 배너 (PendingActionsBanner)
- **조작**: 페이지 상단 배너 확인
- **예상 결과** (건수 > 0일 때):
  - "출고 처리 N건" / "수령 확인 N건" / "입고요청 N건" 등 텍스트 배너
  - 아래에 "대시보드에서 확인" 링크
- ☐ 확인

### B-3. 할일 카드 클릭 → 해당 페이지 이동
- **조작**: 대시보드의 각 할일 카드 클릭
- **예상 결과**:
  - "수령 대기" → `/shipment/dashboard?filter=todo`
  - "출고 처리" → `/shipment/dashboard?filter=todo`
  - "예약판매 대기" → `/sales/entry?tab=preorders`
  - "수평이동" → `/shipment/transfer`
  - "매출 등록" 버튼 → `/sales/entry`
  - 오늘매출/월간매출 카드 → 판매분석 페이지
- ☐ 확인

### B-4. 재고 부족 알림 — 재고요청 보내기
- **조작**: 대시보드 하단 "재고 부족" 테이블에서 "요청" 버튼 클릭
- **예상 결과**:
  - 다른 매장에 재고 있는 상품 옆에 "요청" 버튼 표시
  - 클릭 시 수평이동 요청 모달 또는 알림 발송
  - "다른 매장" 칩에 매장명 + 재고 수량 표시
- ☐ 확인

## C. 매출등록 (가장 중요 — 매일 사용)

### C-1. 매출관리 페이지 진입 — 자동 설정 확인
- **조작**: 판매관리 > 매출관리 클릭
- **예상 결과**:
  - 4개 탭 표시: "매출등록", "판매내역", "고객 반품 관리", "예약판매"
  - "매출등록" 탭이 기본 선택됨
  - 매출일: 오늘 날짜 자동 선택
  - 거래처: 우리 매장이 자동 선택 (Select disabled 상태)
  - 입력 모드 토글: "수동 입력" / "바코드 스캔"
  - 하단에 빈 상품 테이블
- ☐ 확인

### C-2. 수동 입력 — 상품 검색 + 선택
- **조작**: 수동 입력 모드에서 "상품 추가" 검색창에 상품명 또는 SKU 2글자 이상 입력
- **예상 결과**:
  - 드롭다운에 검색 결과 표시 (상품명, SKU, 색상, 사이즈, **재고 수량**)
  - 재고 0인 상품은 빨간색, 재고 있는 상품은 초록색 표시
  - 상품 클릭 → 테이블에 행 추가
- ☐ 확인

### C-3. 바코드 스캔 모드 — 바코드 입력
- **조작**: "바코드 스캔" 모드 선택 → 바코드 입력창에 바코드 번호 입력 + Enter
- **예상 결과**:
  - 해당 바코드의 상품이 테이블에 추가됨
  - 이미 추가된 상품이면 수량 +1 증가
  - 존재하지 않는 바코드 입력 시 "상품을 찾을 수 없습니다" 에러
- ☐ 확인

### C-4. 상품 추가 후 금액 계산 확인
- **조작**: 상품 추가 후 테이블 확인
- **예상 결과**:
  - 유형: "정상" (할인가/행사가 없는 경우) 또는 "할인"/"행사" 자동 결정
  - 수량: 기본 1, 변경 가능 (InputNumber)
  - 단가: 가격 우선순위에 따라 자동 결정 (거래처별 행사가 > 전체 행사가 > 할인가 > 정상가)
  - 소계: 수량 × 단가 (자동 계산)
  - 수량 변경 시 소계 즉시 갱신
- ☐ 확인

### C-5. 정상판매 1건 등록
- **조작**:
  1. 상품 1개 추가 (할인가/행사가 없는 상품)
  2. 수량 확인 (기본 1)
  3. "매출 등록" 버튼 클릭
- **예상 결과**:
  - 성공 메시지: "1건 매출이 등록되었습니다"
  - 유형 컬럼에 "정상" 태그 표시
  - 테이블 초기화 (다음 등록 준비)
- **사전 조건**: 등록 전 재고현황에서 해당 상품 재고 수량 메모해둘 것
- ☐ 확인

### C-6. 할인판매 등록 — 할인가 적용 확인
- **조작**: discount_price가 설정된 상품 추가 후 등록
- **예상 결과**:
  - 유형 자동 "할인" 표시
  - 단가가 base_price가 아닌 discount_price로 표시
  - 등록 후 판매내역 탭에서 유형 "할인" 태그 확인
- ☐ 확인

### C-7. 행사판매 등록 — 행사가 적용 확인
- **조작**: event_price가 설정된 상품 (우리 매장이 event_store_codes에 포함) 추가 후 등록
- **예상 결과**:
  - 유형 자동 "행사" 표시
  - 단가가 event_price로 표시 (할인가/정상가보다 우선)
  - 등록 후 판매내역 탭에서 유형 "행사" 태그 확인
- ☐ 확인

### C-8. 다건 등록 — 여러 상품 동시 등록
- **조작**: 상품 2~3개를 테이블에 추가한 후 한번에 "매출 등록"
- **예상 결과**:
  - "N건 매출이 등록되었습니다" (추가한 상품 수)
  - 판매내역 탭에서 동일 시간대에 N건 표시
- ☐ 확인

### C-9. 등록 후 재고 차감 확인
- **조작**: 재고현황(`/inventory/status`)에서 방금 등록한 상품 검색
- **예상 결과**:
  - 재고 = (등록 전 재고) - (등록 수량)
  - 재고가 0 미만이어도 등록은 성공 (마이너스 재고 허용)
  - 재고 부족 시 경고 메시지가 응답에 포함되었을 수 있음
- ☐ 확인

### C-10. 판매내역 탭에서 등록한 매출 조회
- **조작**: 매출관리 > "판매내역" 탭 클릭 → 오늘 날짜로 조회
- **예상 결과**:
  - 방금 등록한 매출이 목록에 표시
  - 상품명, SKU, 색상, 사이즈, 유형(태그), 수량, 단가, 합계 모두 정확
  - 반품은 금액이 빨간색(음수)으로 표시
- ☐ 확인

### C-11. 매출 수정 — 수량 변경
- **조작**: 판매내역 탭에서 방금 등록한 매출 행의 "수정" 버튼(연필 아이콘) 클릭
- **예상 결과**:
  - 수정 모달 표시: 상품 정보(읽기 전용), 판매유형, 수량(변경 가능), 단가(변경 불가, disabled), 메모
  - 수량을 2 → 3으로 변경 후 저장
  - 성공 메시지 표시
  - 재고현황 확인: 재고가 1개 더 줄어 있어야 함
  - **단가 변경 시도**: 단가 필드가 disabled여서 변경 불가 확인
- ☐ 확인

### C-12. 매출 삭제 — 재고 복구 확인
- **조작**: 판매내역 탭에서 매출 행의 "삭제" 버튼(휴지통 아이콘) 클릭
- **예상 결과**:
  - "정말 삭제하시겠습니까?" 확인 팝업
  - 확인 → 성공 메시지
  - 재고현황 확인: 삭제한 수량만큼 재고 복구
- ☐ 확인

### C-13. 매출 수정/삭제 — 과거 날짜 제한 (매장관리자)
- **조작**: 판매내역 탭에서 어제 또는 이전 날짜 매출의 수정/삭제 시도
- **예상 결과**:
  - 수정 버튼 클릭 → "당일 매출만 수정할 수 있습니다" 에러 (403)
  - 삭제 버튼 클릭 → "당일 매출만 삭제할 수 있습니다. 본사에 문의해주세요." 에러 (403)
  - 오늘 등록한 매출은 수정/삭제 정상 동작
- ☐ 확인

### C-14. 면세 금액 입력
- **조작**: 매출 등록 시 면세(Tax Free) 스위치 ON → 등록
- **예상 결과**:
  - 면세 스위치 ON → 전체 금액의 10% 자동 면세 적용
  - 또는 개별 항목에 T/F(면세금액) 직접 입력 가능
  - 면세금액이 총액의 10% 초과 시 자동으로 10%로 제한 (서버에서 절삭)
  - 판매내역 탭에서 "T/F" 컬럼에 초록색으로 면세 금액 표시
- ☐ 확인

### C-15. 고객 연동 등록
- **조작**: 매출 등록 시 "고객 검색"란에서 고객 이름/전화번호 검색 → 고객 선택 → 등록
- **예상 결과**:
  - 등록 성공 후 고객관리 > 해당 고객 상세에서 구매이력에 방금 매출 자동 반영
  - 구매 횟수/금액 통계 갱신
- ☐ 확인

### C-16. 중복 등록 방지 (5초 룰)
- **조작**: 매출 등록 직후 5초 이내에 동일 매장+날짜로 다시 등록 시도
- **예상 결과**:
  - "중복 등록 감지: 잠시 후 다시 시도해주세요." 에러 메시지
  - 5초 이후 재시도하면 정상 등록
- ☐ 확인

## D. 고객반품 (판매 취소/교환)

### D-1. 고객 반품 탭 진입
- **조작**: 매출관리 페이지 > "고객 반품 관리" 탭 클릭
- **예상 결과**:
  - 기존 반품 목록 테이블 표시 (반품일, 상품명, SKU, 수량, 반품금액(빨간), 반품사유(보라 태그))
  - 필터: 거래처(매장은 고정), 기간, 검색
  - "반품 등록" 버튼 표시 (매니저만)
- ☐ 확인

### D-2. 원본 반품 — 판매내역에서 반품 처리
- **조작**:
  1. 판매내역 탭에서 반품할 매출 행의 "반품" 버튼(롤백 아이콘) 클릭
  2. 반품 모달에서: 원본 수량×단가 표시, 반품 수량 입력, 반품 사유 선택(드롭다운)
  3. 확인
- **예상 결과**:
  - 반품 수량 기본값 = 원본 수량 (전량 반품)
  - 부분 반품 시 수량 줄일 수 있음
  - 반품사유 미선택 시 "반품 사유를 선택해주세요" 에러
  - 성공 시: 반품 목록에 새 건 추가, 금액 빨간색(음수)
  - 재고 확인: 반품 수량만큼 재고 증가
- ☐ 확인

### D-3. 부분 반품 후 추가 반품
- **조작**: 3개 판매한 매출에서 1개만 반품 → 이후 나머지 2개 추가 반품 시도
- **예상 결과**:
  - 반품 가능 수량 조회 시 "잔여 2개" 표시
  - 2개 반품 → 성공
  - 이후 추가 반품 시도 → "반품 가능 수량을 초과합니다" 에러
- ☐ 확인

### D-4. 직접 반품 (원본 매출 없이)
- **조작**: "고객 반품 관리" 탭 > "반품 등록" 버튼 클릭
- **예상 결과**:
  - 모달: 고객 검색, 바코드/SKU 스캔, 상품 검색, 반품 수량, 반품사유(필수)
  - 본사 사용자는 거래처 선택 필수
  - 매장 사용자는 거래처 자동 고정
  - 반품 금액 = 수량 × 정가(base_price) 자동 계산
  - 재고: 반품 수량만큼 증가
- ☐ 확인

### D-5. 30일 반품 기한 제한 (매장관리자)
- **조작**: 판매내역 탭에서 31일 이상 된 매출의 "반품" 버튼 클릭
- **예상 결과**:
  - "판매일로부터 30일이 지난 매출은 본사 승인이 필요합니다." 에러 (403)
  - 30일 이내 매출은 정상 반품 가능
- ☐ 확인

### D-6. 교환 처리
- **조작**: 판매내역 탭에서 매출 행의 "교환" 버튼(스왑 아이콘) 클릭
- **예상 결과**:
  - 교환 모달 표시:
    - 상단: 원본 상품 정보 (읽기 전용, "원본 상품 (반품 처리됨)" 표시, 교환 가능 수량)
    - 교환 사유 (필수)
    - 구분선 "교환 컬러/사이즈 선택": 같은 상품의 다른 컬러/사이즈 드롭다운 (원본 제외, 자동 로드)
    - 수량(최대 교환가능수량), 단가
    - 금액 비교: "반품: -금액" (빨강) / "교환: +금액" (초록) / 차액 (볼드)
  - 성공 시:
    - 원본 상품 재고 +반품수량
    - 교환 상품 재고 -교환수량
    - 판매내역에 반품 + 신규매출 2건 생성
- ☐ 확인

### D-7. 반품 후 원본 매출 삭제 시도
- **조작**: 판매내역 탭에서 반품 처리된 매출의 "삭제" 버튼 클릭
- **예상 결과**: "이 매출에 연결된 반품 N건이 있어 삭제할 수 없습니다. 반품을 먼저 삭제해주세요." 에러
- ☐ 확인

## E. 예약판매

### E-1. 예약판매 탭 진입
- **조작**: 매출관리 페이지 > "예약판매" 탭 클릭
- **예상 결과**:
  - 대기 상태 예약판매 목록 표시
  - 컬럼: 예약일, 상품명, SKU, 수량, 단가, 금액, 상태(대기/해소), 고객
  - 매니저: "삭제" 액션 버튼 표시 (해소는 입고/수령 시 자동 처리)
  - 직원: 목록 조회만 가능
- ☐ 확인

### E-2. 예약판매 등록 (재고 부족 상품)
- **조작**: 매출등록 탭에서 재고 0인 상품 추가 → 매출 등록
- **예상 결과**:
  - 재고 부족이어도 매출 등록 성공 (마이너스 재고 허용)
  - 또는 예약판매 탭에 "대기" 상태로 등록 (시스템 설정에 따라)
  - warnings 응답에 "재고 부족" 경고 포함 가능
- ☐ 확인

### E-3. 예약판매 자동 해소 확인
- **조작**: 예약판매 대기 중인 상품이 입고확정 또는 출고수령(RECEIVED)되면 확인
- **예상 결과**:
  - 수령 완료 후 해당 상품의 예약판매가 FIFO 순서로 자동 해소
  - 해소 시 가격은 해소 시점의 행사가/할인가/정상가 자동 결정
  - 해소된 매출은 판매내역 탭에 정상 표시
  - (수동 해소 버튼은 없음 — 자동 해소만 지원)
- ☐ 확인

### E-4. 예약판매 삭제
- **조작**: "대기" 상태 예약판매의 "삭제" 버튼 클릭
- **예상 결과**:
  - 확인 팝업 → 삭제 성공
  - 이미 "해소"된 건은 삭제 불가
  - 매장직원(STORE_STAFF)은 삭제 불가 (403)
- ☐ 확인

## F. 판매내역 (매출관리 > 판매내역 탭)

### F-1. 판매내역 탭 — 기본 조회
- **조작**: 매출관리 > "판매내역" 탭 클릭
- **예상 결과**:
  - 필터: 기간(기본 오늘), 검색(SKU/상품명)
  - 거래처 필터: 매장 사용자는 자기 매장 고정 (변경 불가, 필터 미표시)
  - 테이블 컬럼: 매출일, 매출번호, SKU, 상품명, 색상, 사이즈, 유형, T/F(면세), 반품사유, 메모, 수량, 단가, 합계
  - 반품 건은 합계가 빨간색(음수)
  - 유형 색상: 정상(파랑), 할인(빨강), 행사(주황), 반품(보라), 예약판매(주황태그), 수정(주황)
- ☐ 확인

### F-2. 날짜 변경 → 데이터 조회
- **조작**: 기간 필터를 지난달로 변경 → "조회" 클릭
- **예상 결과**:
  - 해당 기간의 매출 목록만 표시
  - 다른 매장 데이터는 표시되지 않음
  - 페이지네이션: 50건 단위, 하단에 "총 N건" 표시
- ☐ 확인

### F-3. 매출 금액 합계 검증
- **조작**: 오늘 매출을 모두 조회하여 합계 계산
- **예상 결과**:
  - 각 행의 합계 = 수량 × 단가
  - 전체 합산 금액이 대시보드의 "오늘 매출" 카드 금액과 일치
- ☐ 확인

### F-4. 매출 액션 버튼 상태 확인
- **조작**: 판매내역 탭의 "관리" 컬럼 버튼들 확인
- **예상 결과**:
  - **매니저 이상만** "관리" 컬럼 표시 (STORE_STAFF는 미표시)
  - **오늘 매출**: 수정(연필), 교환(스왑), 반품(롤백), 삭제(휴지통) 활성
  - **과거 매출(매장관리자)**: 수정/삭제 비활성 (당일만 가능), 교환/반품은 30일 이내 가능
  - **반품 건**: 매장관리자는 "-" 표시 (삭제 불가), 본사만 삭제 가능
  - **수정기록 건**: "수정기록" 텍스트만 표시 (액션 없음)
  - **예약판매 건**: "예약판매 탭에서 관리" 태그 표시 (액션 없음)
- ☐ 확인

## G. 출고 수령 (본사 → 매장)

### G-1. 종합출고관리 — 우리 매장 출고 목록
- **조작**: 출고관리 > 종합출고관리 클릭
- **예상 결과**:
  - 4개 섹션: 출고(본사→매장), 반품(매장→본사), 수평이동(매장↔매장), 출고요청(매장→본사)
  - 각 섹션에 상태별 칩(PENDING/SHIPPED/RECEIVED 등) + 건수
  - 상단 "해야할일" / "대기중" 2개 요약 카드
  - 우리 매장 관련 건만 표시 (출발 또는 도착이 우리 매장)
- ☐ 확인

### G-2. SHIPPED 출고 → 수령확인 버튼
- **조작**: "출고(본사→매장)" 섹션에서 SHIPPED 상태 칩 클릭 → 해당 건의 "수령확인" 버튼 클릭
- **예상 결과**:
  - 수령확인 버튼은 도착지가 우리 매장인 건에만 표시 (초록색 아이콘)
  - 클릭 시 ReceivedQtyModal 열림
  - 모달 내용: 의뢰번호, 유형, 출발지, 요청일
  - 품목별: SKU, 상품명, 색상, 사이즈, 출고수량(주황), 수령수량(입력), 차이
- ☐ 확인

### G-3. 수령수량 동일 입력 → RECEIVED
- **조작**: ReceivedQtyModal에서 모든 품목의 수령수량 = 출고수량으로 입력 → 확인
- **예상 결과**:
  - 상태 SHIPPED → RECEIVED (초록 태그)
  - 성공 메시지
  - 수령수량 컬럼에 초록색 숫자 표시
- ☐ 확인

### G-4. 수령 후 재고 증가 확인
- **조작**: 재고현황에서 수령한 상품 검색
- **예상 결과**: 재고 = (수령 전) + (수령 수량)
- ☐ 확인

### G-5. 수령수량 불일치 → DISCREPANCY
- **조작**: ReceivedQtyModal에서 일부 품목의 수령수량을 출고수량보다 적게 입력 → 확인
- **예상 결과**:
  - 상태 SHIPPED → DISCREPANCY (주황 태그)
  - 차이 컬럼에 빨간색 표시 (출고 5 → 수령 3 = 차이 -2)
  - "수량재확인" 버튼이 표시됨
  - 재고는 실수령량만큼만 증가
  - 관리자(본사)만 "완료처리" 가능
- ☐ 확인

### G-6. 다른 매장 출고건 격리 확인
- **조작**: 출고 목록에서 다른 매장 간 출고건이 보이는지 확인
- **예상 결과**: 출발 또는 도착이 우리 매장인 건만 표시. 다른 매장↔다른 매장 건은 미표시
- ☐ 확인

### G-7. 매장 사용자 취소 제한 확인
- **조작**: SHIPPED 상태 출고건의 "취소" 버튼 확인
- **예상 결과**: SHIPPED/DISCREPANCY 단계에서 매장 사용자는 취소 버튼 비활성 (본사만 취소 가능)
- ☐ 확인

## H. 입고요청 (매장 → 본사)

### H-1. 입고요청 페이지 진입
- **조작**: 입고관리 > 종합입고관리 또는 직접 `/shipment/store-request` 접속
- **예상 결과**:
  - 6개 상태 카드: 요청중(PENDING), 출고완료(SHIPPED), 수량불일치(DISCREPANCY), 수령완료(RECEIVED), 거절(REJECTED), 취소(CANCELLED)

### H-2. 새 입고요청 등록
- **조작**: "입고요청" 버튼 클릭 → 모달에서 상품 검색/추가 → 수량 입력 → 저장

- ☐ 확인

### H-3. 요청 목록 확인 + 상세
- **조작**: 등록한 요청 행의 "상세" 버튼 클릭
- **예상 결과**:

  - 상태: PENDING, 출고/수령수량은 아직 0
- ☐ 확인

### H-4. PENDING 요청 취소
- **조작**: PENDING 상태 요청의 "취소" 버튼 클릭
- **예상 결과**:
  - 확인 팝업 → 취소 성공 → 상태 CANCELLED
  - PENDING 상태에서만 취소 가능 (요청자 본인만)
  - 수량불일치(DISCREPANCY) 상태에서도 요청자 본인 취소 가능
- ☐ 확인

### H-5. 본사 출고 후 상태 변경 확인
- **조작**: 본사에서 해당 요청을 출고 처리한 후 매장에서 확인
- **예상 결과**:
  - 상태 PENDING → SHIPPED
  - "수령확인" 버튼 활성화
  - 출고수량 컬럼에 주황색 숫자 표시
- ☐ 확인

### H-6. 입고요청 수령확인
- **조작**: SHIPPED 상태 요청의 "수령확인" 버튼 클릭 → 수령수량 입력 → 확인
- **예상 결과**: G-3~G-5와 동일 (RECEIVED 또는 DISCREPANCY)
- ☐ 확인

## I. 수평이동 (매장 ↔ 매장) — 매장관리자만

### I-1. 수평이동 페이지 — 대시보드 카드
- **조작**: 출고관리 > 수평이동 클릭
- **예상 결과**:
  - 5개 상태 카드: 대기(PENDING, 노란), 이동중(SHIPPED, 파랑), 수량불일치(DISCREPANCY, 주황), 완료(RECEIVED, 초록), 취소(CANCELLED, 빨강)
  - 각 카드에 건수 + "처리 필요" 뱃지 (우리가 해야할 건)
  - 테이블에 "구분" 컬럼: "보내기"(파란 태그) / "받기"(초록 태그)
- ☐ 확인

### I-2. 보내기 — 수평이동 등록
- **조작**: "수평이동 등록" 버튼 → 모달에서:
  1. 이동 출발: 우리 매장 (자동 고정)
  2. 이동 도착: 다른 매장 선택
  3. 상품 검색/추가 (또는 "전체재고" 버튼)
  4. 수량 입력
  5. 등록
- **예상 결과**:
  - 도착매장 = 우리 매장 선택 시: 에러 (같은 매장 이동 불가)
  - 상품 추가 시 현재 재고 표시 (빨강=0, 초록=있음)
  - 등록 성공 → PENDING 상태 + "보내기" 태그
  - **재고 변동 없음** (PENDING이므로)
- ☐ 확인

### I-3. 출고확인 (PENDING → SHIPPED)
- **조작**: 우리가 보낸(출발=우리매장) PENDING 건의 "출고확인" 버튼 클릭
- **예상 결과**:
  - ShippedQtyModal 열림: 품목별 출고수량 입력
  - 출고수량 ≤ 의뢰수량 검증
  - 확인 → 상태 SHIPPED
  - **우리 매장 재고 차감** (출고수량만큼)
  - 재고현황 확인: 재고 감소
- ☐ 확인

### I-4. 받기 — 수령확인 (SHIPPED → RECEIVED)
- **조작**: 상대 매장이 보낸(도착=우리매장) SHIPPED 건의 "수령확인" 버튼 클릭
- **예상 결과**:
  - ReceivedQtyModal 열림: 품목별 수령수량 입력
  - 출고수량 대비 수령수량 차이 표시 (초록=일치, 빨강=불일치)
  - 전량 일치 → RECEIVED
  - **우리 매장 재고 증가** (수령수량만큼)
- ☐ 확인

### I-5. 수령수량 불일치 → DISCREPANCY
- **조작**: 수령수량을 출고수량과 다르게 입력 후 확인
- **예상 결과**:
  - 상태 SHIPPED → DISCREPANCY
  - 실수령량만큼만 재고 증가
  - "수량재확인" 버튼 표시
  - 관리자만 "완료처리" 가능
- ☐ 확인

### I-6. 같은 매장 이동 시도 → 에러
- **조작**: 등록 모달에서 출발=우리매장, 도착=우리매장으로 선택 후 등록
- **예상 결과**: "같은 매장으로 이동할 수 없습니다" 에러
- ☐ 확인

### I-7. PENDING 취소 → 재고 원복
- **조작**: PENDING 상태 건의 "취소" 버튼 클릭
- **예상 결과**:
  - PENDING 취소: 재고 변동 없음 (원래 없었으므로)
  - 상태 → CANCELLED
- ☐ 확인

### I-8. SHIPPED 취소 제한
- **조작**: SHIPPED 상태 건의 "취소" 버튼 확인
- **예상 결과**: 매장 사용자는 SHIPPED 이후 취소 불가 (본사만 가능). 취소 버튼 비활성 또는 미표시
- ☐ 확인

### I-9. 상태 필터 + 검색
- **조작**: 상태 카드 클릭 → 해당 상태만 필터, 검색창에 상품명 입력
- **예상 결과**: 해당 상태 건만 테이블에 표시, 검색 결과 정확
- ☐ 확인

### I-10. 펼침 행 — 품목 상세
- **조작**: 테이블 행 왼쪽 화살표 클릭
- **예상 결과**: 품목별 SKU, 상품명, 색상, 사이즈, 의뢰수량, 출고수량, 수령수량 상세 표시
- ☐ 확인

## K. 재고관리

### K-1. 재고현황 — 우리 매장 재고
- **조작**: 재고관리 > 재고현황 클릭
- **예상 결과**:
  - 거래처 필터: 우리 매장 자동 선택 (다중 선택 가능)
  - 테이블: 거래처, 상품코드, 상품명, 카테고리, 시즌, SKU, 색상, 사이즈, 판매가, 재고

  - 통계 카드 (STORE_MANAGER): "총 재고수량", "보유 매장", "재입고 관리(바로가기)", "품절" 4개 + 음수 재고 있으면 "예약판매(음수)" 추가
  - 추가 필터: 카테고리, 연도(부터/까지), 시즌, 색상, 사이즈, 상태

  - 조회만 가능 (수정 불가)
- ☐ 확인

### K-4. 재입고 추천
- **조작**: 재고관리 > 매장 재입고 추천 클릭
- **예상 결과**:

  - 일부 사이즈만 빠진 품목 목록: 품번, 상품명, 카테고리, 시즌, 컬러, 빠진 사이즈(빨간 태그), 보유 사이즈, 전체

  - 매장 자동 선택 (거래처 선택 불가), 조회만 가능
- ☐ 확인

### K-5. 재고 수량 실시간 반영

- **예상 결과**: 매출/출고/수령 즉시 재고 수량에 반영 (페이지 새로고침 필요할 수 있음)
- ☐ 확인

### K-6. 재고 조정 메뉴 접근 제한
- **조작**: URL 직접 입력 `/inventory/adjust` 시도
- **예상 결과**: 메뉴에는 미노출 (ADMIN/HQ 전용 메뉴). STORE_MANAGER는 URL 접근 시 재고현황 페이지 동일 표시. STORE_STAFF는 접근 차단
- ☐ 확인

## M. 바코드

  - **원가(cost_price) 미노출**
- ☐ 확인

### M-2. 바코드로 상품 검색 (스캔)

- **예상 결과**:
  - 스캔 결과 카드: 상품명, 상품코드, SKU, 카테고리, 컬러/사이즈, 바코드(등록=초록/미등록=빨강), 판매가, 매장 재고
  - 매장 재고 색상: 0이하=빨강, 1~5=주황, 6+=초록
  - 최근 스캔 히스토리: 최대 10건, 가로 스크롤 카드 형태
  - 존재하지 않는 바코드/SKU 입력 시 "상품을 찾을 수 없습니다" 경고
- ☐ 확인

## N. 고객관리 (CRM) — 매장관리자만

### N-1. 고객 대시보드 — 우리 매장 고객만 표시
- **조작**: 고객관리 메뉴 클릭 (`/crm`)
- **예상 결과**:
  - 매장 사용자 기본화면: **CRM 대시보드** (통계카드 + 등급분포 차트 + RFM 세그먼트 + LTV 상위고객 + 생일고객)
  - 우리 매장 고객만 표시 (서버측 `getStorePartnerCode` 필터)
  - 빠른 검색: 이름/전화번호 2자 이상 입력 → 디바운스(300ms) 검색 결과 테이블
  - 전체 목록 + 등급 필터(전체/VVIP/VIP/일반/신규)는 `/crm/list` 에서 제공
  - HQ 사용자 기본화면: **매장별 고객 분포** (CrmStoreData)
- ☐ 확인

### N-2. 고객 등록
- **조작**: "고객 등록" 버튼 → 이름, 전화번호, 이메일 등 입력 → 저장
- **예상 결과**:
  - partner_code가 우리 매장으로 자동 설정
  - 중복 전화번호 → "이미 등록된 전화번호입니다" (409)
  - 이름/전화번호 필수 (미입력 시 400)
- ☐ 확인

### N-3. 고객 상세 — 9개 탭 + 스타일 노트
- **조작**: 고객 행 클릭 → 상세 페이지 (`/crm/:id`)
- **예상 결과**:
  - 헤더: 고객명 + 등급 태그 + 수정/삭제 버튼
  - 기본정보 카드: 이름, 전화, 이메일, 성별, 생년월일, 매장, 등록일, 수신동의, 주소, 메모
  - 스타일 노트 카드: 선호 사이즈/스타일/색상/체형 메모 (수정 가능)
  - 통계 카드 4개: 총 구매액, 구매횟수, 평균 구매액, 최근 구매일
  - **9개 탭**: 구매이력(purchases), 구매패턴(patterns), 방문이력(visits), 상담이력(consultations), 피드백(feedback), 메시지이력(messages), 등급이력(tierHistory), 배송추적(shipments), RFM분석(rfm)
  - 태그/플래그 관리: 인라인 추가/삭제
- ☐ 확인

### N-4. 고객 등급 확인
- **조작**: 고객 상세에서 등급(tier) 정보 확인
- **예상 결과**: 현재 등급 표시, 등급 변경 이력 조회 가능
- ☐ 확인

### N-5. 다른 매장 고객 접근 차단
- **조작**: 다른 매장 고객의 customer_id를 URL에 직접 입력 (예: `/crm/999`)
- **예상 결과**: "다른 매장의 고객 정보에 접근할 수 없습니다." (403)
- ☐ 확인

### N-6. 고객 수정/삭제
- **조작**: 고객 상세에서 정보 수정 → 저장 / 삭제 버튼 클릭
- **예상 결과**:
  - 수정: 우리 매장 고객만 수정 가능
  - 삭제: 소프트 삭제 (is_active=false) → 목록에서 사라짐
  - 다른 매장 고객의 수정/삭제 시도 → 403
- ☐ 확인

## P. 직원 관리 — 매장관리자만

### P-1. 직원 목록 — 우리 매장 직원만
- **조작**: 직원관리 메뉴 클릭
- **예상 결과**:
  - 페이지 제목: "직원 관리"
  - 우리 매장 소속 STORE_STAFF 역할 직원만 표시
  - 다른 매장 직원, 본사 직원 미표시
  - 컬럼: 아이디, 이름, 직급(태그), 소속(태그), 상태(활성/비활성), 최종로그인, 관리
  - "직원 등록" 버튼 표시
- ☐ 확인

### P-2. 직원 등록
- **조작**: "직원 등록" → 아이디, 이름, 비밀번호(4자+), 역할 선택 → 저장
- **예상 결과**:
  - 역할 선택: STORE_STAFF만 선택 가능 (자기보다 하위 역할만)
  - 소속: 우리 매장 자동 설정
  - 저장 성공 → 목록에 추가
  - 해당 계정으로 로그인 가능
- ☐ 확인

### P-3. 직원 수정/삭제
- **조작**: 직원 행의 수정/삭제 버튼
- **예상 결과**:
  - 수정: 이름, 비밀번호 변경 가능
  - 삭제: 소프트 삭제 (비활성 처리)
  - 자기 자신은 삭제 불가
  - 동일/상위 역할 직원은 삭제 불가
- ☐ 확인

### P-4. 다른 매장 직원 미표시
- **조작**: 직원 목록에서 다른 매장 직원이 보이는지 확인
- **예상 결과**: 우리 매장의 STORE_STAFF만 표시. 다른 매장/본사 직원은 절대 미표시
- ☐ 확인

## Q. 매장직원(STORE_STAFF) 전용 확인

> **테스트 환경**: `npm run dev:staff` (포트 5175) 또는 위에서 등록한 STORE_STAFF 계정으로 로그인

### Q-1. 매출등록 가능
- **조작**: 판매관리 > 매출관리 → 매출등록 탭 → 상품 추가 → 등록
- **예상 결과**: 정상 등록 성공 (STORE_STAFF도 매출 등록 가능)
- ☐ 확인

### Q-2. 바코드 스캔 가능 (대시보드 데이터 제한)
- **조작**: 바코드관리 진입 → 스캔 입력창에 바코드/SKU 입력
- **예상 결과**: 스캔 검색은 정상 작동. 단, 통계 카드/상품 목록은 로드 불가 (서버에서 STORE_STAFF 제외)
- ☐ 확인

### Q-3. 매출 수정/삭제 불가
- **조작**: 판매내역 탭에서 수정/삭제 버튼 확인
- **예상 결과**:
  - 수정 버튼 클릭 → "권한이 없습니다" (403)
  - 삭제 버튼 클릭 → "권한이 없습니다" (403)
  - 또는 버튼 자체가 미표시
- ☐ 확인

### Q-4. 반품/교환 불가
- **조작**: 판매내역 탭에서 반품/교환 버튼 확인
- **예상 결과**: 반품/교환 버튼 미표시 또는 클릭 시 403
- ☐ 확인

### Q-5. 출고관리 메뉴 접근 불가

### Q-6. 수평이동 메뉴 미표시

- ☐ 확인

### Q-7. 재고 메뉴 접근 제한

- **예상 결과**: 재고현황/매장별재고/재고조정/재고처리 메뉴 모두 미표시

## R. 데이터 격리 확인 (보안)

### R-1. 매출 격리 — 다른 매장 매출 미표시
- **조작**: 판매내역 탭에서 전체 기간 조회
- **예상 결과**: 우리 매장 매출만 표시. 다른 매장 매출 건수 0
- ☐ 확인

### R-2. 재고 격리 — 다른 매장 재고 수정 불가
- **조작**: 재고현황에서 다른 매장 재고 확인
- **예상 결과**:
  - 매장별 재고 모달에서 다른 매장 재고 "조회"는 가능
  - 재고 조정/수정 기능은 접근 불가 (ADMIN/HQ만)
- ☐ 확인

### R-3. 출고 격리 — 다른 매장 출고건 미표시
- **조작**: 출고조회에서 전체 조회
- **예상 결과**: 출발 또는 도착이 우리 매장인 건만 표시
- ☐ 확인

### R-4. 고객 격리 — 다른 매장 고객 접근 불가
- **조작**: 고객관리에서 전체 조회 + 다른 매장 고객 ID URL 직접 접속
- **예상 결과**:
  - 목록: 우리 매장 고객만
  - URL 직접 접속: "다른 매장의 고객 정보에 접근할 수 없습니다." (403)
- ☐ 확인

### R-5. 관리자 페이지 URL 직접 접근 차단
- **조작**: 아래 URL을 직접 브라우저에 입력:
  - `/system/settings` (시스템 설정) — ADMIN_SYS 전용
  - `/inventory/transactions` (재고변동내역) — ADMIN 전용
  - `/fund` (자금관리) — ADMIN 전용
  - `/production` (생산기획) — ADMIN 전용
- **예상 결과**: 모두 403 페이지 또는 리다이렉트
- **참고**: `/inventory/adjust`는 라우트가 ADMIN_HQ_STORE이므로 STORE_MANAGER는 접근 가능 (재고현황 페이지와 동일). 단, 실제 재고조정 API(`POST /adjust`)는 ADMIN/HQ만 호출 가능
- ☐ 확인

### R-6. API 직접 호출 차단
- **조작**: 브라우저 개발자도구(F12) > Console에서:
  ```js
  fetch('/api/inventory/transactions').then(r => r.json()).then(console.log)
  fetch('/api/funds').then(r => r.json()).then(console.log)
  fetch('/api/productions').then(r => r.json()).then(console.log)
  ```
- **예상 결과**: 모두 `{ error: "..." }` + status 403 반환
- ☐ 확인

## S. 입고관리 — 출고불일치 재확인

### S-1. 입고관리 > 출고불일치 탭
- **조작**: 입고관리 > 종합입고관리 → "출고불일치" 탭 (매장 사용자에게만 표시)
- **예상 결과**:
  - "수량불일치 건 — 본사 출고수량과 매장 수령수량이 다른 건입니다" 안내 표시
  - DISCREPANCY 상태 건 목록: 의뢰번호, 유형, 출발, 요청일, 요청수량, 출고수량(주황), 수령수량(빨강), 메모
  - "수량재확인" 버튼 표시
- ☐ 확인

### S-2. 수량재확인
- **조작**: "수량재확인" 클릭 → ReceivedQtyModal에서 수량 수정 → 확인
- **예상 결과**:
  - 수령수량을 정정 가능
  - 출고수량과 수령수량 차이를 초록(일치)/빨강(불일치)으로 표시
  - 재확인 후에도 불일치면 DISCREPANCY 유지 (본사가 최종 완료 처리)
- ☐ 확인
